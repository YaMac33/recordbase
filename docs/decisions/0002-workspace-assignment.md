# ADR-0002: ワークスペース割り当て方針（1ユーザー=1ワークスペース ＋ workspace_members）

- ステータス: 採択
- 日付: 2026-06-13
- 関連: DESIGN.md ADR-03（テナント＝シングル想定＋`workspace_id`）, §11（マルチテナント昇格手順）, ADR-05（Better Auth）

## 背景
全テーブルに `workspace_id` を持たせ、後からマルチテナントへ「昇格」する方針（ADR-03）は確定済み。
フェーズ2（認証）でユーザーをサインアップさせる際、**ユーザーをどのワークスペースに割り当てるか**を
決める必要がある。候補は次の2つ。

- **A. 1ユーザー = 1ワークスペース**：サインアップ時に専用 ws を自動生成して本人を所属させる。
- **B. 共有ワークスペース**：全ユーザーを単一の共通 ws に所属させる。

## 決定
**A を採用する。** さらに将来の共同作業に備え、最初から **`workspace_members`（user_id × workspace_id × role）**
を用意し、サインアップ時に「専用 ws ＋ 本人を `owner` とする membership 1件」を作成する。

## 理由
- 個人向けオンライン SaaS では、テナント境界＝データ可視性の境界。A は他人のデータが原理的に見えず、
  `workspace_id` スコープのバグがあっても被害が自 ws 内に閉じる。B は一般公開時点で境界が無く不適。
- A は §11 の「`workspace_id` 前提を活かした段階的マルチテナント昇格」と前進方向で一致する。
  B から A への移行は既存レコードの ws 振り分け（データ移行）が必要で逆行する。
- 課金・退会・データ削除の単位を ws に一致させられる。
- `workspace_members` を最初から置くことで、A の分離性を保ったまま、将来は「招待＋membership 追加」
  だけで共有・共同編集へ拡張でき、B の利点（共同作業）も後付けで取り込める。

## 帰結
- スキーマに `workspace_members` を追加する（role は当面 `owner`/`admin`/`member` を想定、値検証は
  `packages/shared` の Zod 側で行う）。
- ユーザー ID は Better Auth が発行する文字列 ID を一次とする。フェーズ1ではドメイン表に
  `user_id` / `created_by` を **テキスト列**として持たせ、Better Auth のユーザーテーブル確定後
  （フェーズ2）に外部キー制約を付与する。
- サインアップフック（専用 ws ＋ owner membership 作成）はフェーズ2で実装する。
- 現時点ではシングルユーザー運用だが、データモデルはマルチテナント前提で一貫させる。

## 代替案（不採用）
- **B. 共有ワークスペース**：実装は最軽量だが、一般公開時にテナント境界が無く、権限レイヤに境界を
  肩代わりさせるため漏洩リスクが高い。後からの分離移行コストも大きく不採用。
