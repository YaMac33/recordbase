# ノーコードアプリ作成ツール — 設計書（オンライン版）

## 0. このドキュメントの位置づけ
- 本書は「設計母艦」の成果物。ここで確定した方針を、VS Code 上の Claude Code が実装する際の一次情報とする。
- 現フェーズ：**設計確定。アプリ本体のコードは未着手。**
- 旧引き継ぎサマリ（LGWAN／オフライン前提・庁内PC1台サーバー）は本書で上書きされた。前提は「オンライン・一般向け SaaS（個人開発）」に変更済み。

## 1. 目的とスコープ
- kintone 型のノーコードアプリ作成ツールを自作する。
- 「1つの DB を複数アプリで共有」「1つのアプリで複数 DB をまたぐ」を成立させる、**データ層とアプリ層を分離したメタ駆動モデル**が中核。
- 一般向けオンライン SaaS として提供する。

## 3. 決定記録（ADR ダイジェスト）
| ID | 決定 | 理由 |
|---|---|---|
| ADR-01 | データの持ち方＝**汎用 JSON ＋昇格** | レコードは JSON で保管し、よく検索／集計する項目だけ生成列・インデックスへ「昇格」。メタ駆動の自由度を保ちつつ性能を後付け補強し、エンジンを作り直さない。 |
| ADR-02 | DB＝**PostgreSQL（JSONB）** | オンライン・複数アクセス前提で SQLite から繰り上げ。JSONB＋GIN／式インデックスで昇格戦略を実装できる。 |
| ADR-03 | テナント＝**シングル想定＋`workspace_id`** | 全テーブルに `workspace_id` を最初から付与。後からマルチテナントへ「昇格」でき、初期コストを抑える。 |
| ADR-04 | スタック＝**Node/TypeScript フルスタック（TS API＋React）** | メタ定義→API→フォーム描画までを型として一本共有。「型がデータ」の設計と相性が良く、1言語で開発でき Claude Code の効率も最大。 |
| ADR-05 | 認証＝**Better Auth（自ホスト）** | TS 製・自ホスト・組織／マルチテナントのプラグインあり。自前実装の危険を避けつつユーザーを自分の DB に保持。per-user 課金なし。 |
| ADR-06 | ORM＝**Drizzle** | SQL-first で JSONB／生成列／式インデックスを直接制御でき、昇格戦略と相性が良い。スキーマが TS で「型がデータ」と一致。 |
| ADR-07 | API フレームワーク＝**Fastify** | 単一サービス API・スキーマ検証・成熟したエコシステム。サーバー常駐型 PaaS 向き。 |
| ADR-08 | ホスティング＝**マネージド PaaS＋マネージド Postgres（Render 想定）** | GitHub push→自動デプロイで運用を借りる。データ用途なので PITR バックアップを重視。代替は Railway。 |

## 4. アーキテクチャ：メタ駆動 3 層
- **データベース層**：テーブル定義＋レコード（業務データの真実の置き場）
- **アプリ層**：そのデータに対する「フォーム・一覧・権限・集計」というビュー定義
- **リレーション層**：DB 間の参照（lookup／関連レコード）

テーブル定義・フィールド定義・アプリ定義・権限を「メタデータ」として DB に持つことで、項目の自由追加と DB の共有を成立させる。

## 5. データの持ち方（ADR-01／02 詳細）
- `records` は汎用テーブル。本体は `data`（JSONB）に保管する。
- 検索・集計・ソートで多用するフィールドのみ、メタ定義をもとに**生成列＋インデックス（または式インデックス）へ昇格**する。
- 物理テーブル動的生成方式は、特定の超高頻度・大量レコードの基幹 DB が出てきた「その時だけ」の例外手段として温存する。

## 6. データモデル（スケッチ）
※ 実装前のスケッチ。型・制約の詳細は実装時に Drizzle スキーマで確定する。全テーブルに `workspace_id`（テナントキー）を持たせる。

| テーブル | 主な列 | 役割 |
|---|---|---|
| `workspaces` | id, name | テナント（当面は単一） |
| `users` | id, workspace_id, email … | 利用者。Better Auth 管理テーブルと連携 |
| `databases` | id, workspace_id, name | データ定義（業務 DB） |
| `fields` | id, database_id, key, type, options(JSON) | 項目定義 |
| `records` | id, workspace_id, database_id, data(JSONB), revision, created_at, updated_at, created_by | 業務データ本体 |
| `apps` | id, workspace_id, name | アプリ（ビューの束） |
| `app_views` | id, app_id, kind(form/list), config(JSON) | フォーム・一覧などの定義 |
| `app_databases` | app_id, database_id | 1アプリ ↔ 複数 DB の対応 |
| `relations` | id, from_field_id, to_database_id, kind(lookup/related) | DB 間参照 |
| `permissions` | id, app_id, scope, role … | 権限（当面アプリ単位＋作成者ベース） |

## 7. 技術スタック（確定）
- **言語**：TypeScript（フロント・バック共通）

- **DB**：PostgreSQL（JSONB）。
- **ホスティング**：Render（GitHub 連携・managed Postgres・PITR）。代替 Railway。

## 9. 競合制御（楽観ロック）
- **方式**：レコード単位の楽観ロック。バージョン判定は整数 `revision` を使う。`updated_at` は表示・監査用とし、衝突判定には使わない（時刻衝突・時計ずれを避けるため）。
- **読み取り**：編集のためにレコードを読むと、クライアントはその時点の `revision`（= N）を保持する。
- **保存（条件付き更新）**：クライアントは編集後データと基準 `revision`（N）を送る。サーバは `id = :id AND workspace_id = :ws AND revision = :N` を条件に UPDATE し、`revision` を N+1 に、`updated_at` を現在時刻へ更新する。
  - 影響行数 **1** → 成功。新しい `revision`（N+1）を返す。
  - 影響行数 **0** → 衝突。レコードを再読込し、存在して `revision` ≠ N なら「他者が更新」、存在しなければ「削除済み」と判定して、それぞれに応じた応答を返す。
- **衝突時の方針（段階導入）**：
  1. 第1段階（堅実）：409 を返し、最新のサーバ版（`revision` ＋ `data`）を同梱。クライアントは読み直して変更を再適用する。
  2. 第2段階（昇格）：`data` が JSON である利点を使い、3-way マージ（基準版・自分の変更・相手の最新）を行い、**同じフィールドを双方が変更した時だけ**本当の衝突として扱う。項目数の多い本ツールでは体験が大きく向上する。基準版はクライアントが読み込み時に保持しているものを使う。
- **新規作成**：挿入なので衝突はないが、二重送信（ダブルクリック等）対策として、クライアント生成のレコード ID（冪等キー）＋ユニーク制約で再送時の重複を防ぐ。
- **削除**：`revision` 条件付き削除で、古い版を見て新しい版を誤削除する事故を防ぐ。将来、`deleted_at` による soft-delete を導入して監査・復元に対応（クエリは未削除のみを対象にスコープ）。
- **範囲**：単一行の条件付き UPDATE は DB レベルで原子的。リレーション等で複数行を同時更新する場合のみトランザクションで囲む。

## 10. DB 間参照（lookup／関連レコード）
- **フィールド型を 3 種で表現する**：
  - **参照（reference）**：他 DB のレコードを指す。JSON には**相手レコードの ID のみ**を保持する（値のコピーは持たない）。単一参照と複数参照（ID 配列）の両方を許す。
  - **ルックアップ（lookup）**：参照フィールドを経由して相手レコードの値を表示するための**派生値**。独立したデータとしては保存せず、読み取り時に解決する。
  - **関連レコード（related）**：参照の逆方向（1 対多）。相手 DB から「このレコードを参照している行」を引いて一覧表示する。保存データは持たない。
- **メタ定義の置き場**：相手 DB（`target_database_id`）、表示に使う項目、単一/複数、参照と関連の別は、すべて `fields.options`（メタ）側に定義する。レコードの JSON は ID だけを持つ。→ メタ駆動で一貫し、JSON を軽く保つ。
- **解決（読み取り）**：
  - ルックアップ・関連は読み取り時に解決するため常に最新。
  - 一覧描画では行ごとに引かず、参照先 DB ごとに ID をまとめて `IN (...)` で一括解決し、N+1 を避ける。
  - 解決クエリは常に `workspace_id` でスコープする（相手 DB も同一ワークスペース内に限定）。
- **整合性（ソフト参照）**：参照 ID は JSON 内にあるため DB の外部キーは効かない。方針は 2 段：
  1. 書き込み時に「参照先が存在し、同一ワークスペースである」ことをアプリ層で検証する。
  2. 参照されているレコードの削除はデフォルトで**禁止（restrict）**。relation ごとに restrict／set null などのポリシーを切り替え可能にする。
- **性能・整合性の昇格**：よく使う参照だけ、参照 ID を**生成列へ昇格→インデックス**し、結合を高速化する。さらに必要なら、その生成列に**本物の外部キー制約**を付けて DB レベルの整合性まで引き上げる。重いルックアップ値だけを非正規化キャッシュする選択肢も、特定箇所に限って後付けできる（いずれも §5 の昇格戦略の一部）。
